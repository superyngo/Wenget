  評估報告

  現有問題分析

  現有的 bucket create 是一個互動式精靈(wizard)，需要用戶一步步輸入資訊。這與 Python 版本的自動化邏輯完全不同：

  | 面向     | 現有 Rust 實現       | Python 實現               |
  |----------|----------------------|---------------------------|
  | 模式     | 互動式 wizard        | 批次自動化                |
  | 來源     | 手動輸入             | 讀取文本文件 + GitHub API |
  | 平台偵測 | 手動輸入             | 自動從 filename 分析      |
  | scripts  | 手動輸入 script type | 自動從Gist/raw URL 取得   |

  新設計提案評估

  您的需求完全合理，以下是詳細評估：

  1. 新CLI 結構

  wenget bucket create [OPTIONS]
      -r, --repos-src<FILE>       #讀取 repos URL 列表 (支援逗號分隔多檔)
      -s, --scripts-src <FILE>     # 讀取 scripts URL 列表 (支援逗號分隔多檔)
      -d, --direct<URL/PATH>,...  # 直接加入 binary/script URL 或本地路徑
      -o, --output <FILE>          # 輸出 manifest.json (預設: manifest.json)
      -t, --token <TOKEN>          # GitHub token (可選，也支援 GITHUB_TOKEN env)

  2. 需要移植的核心邏輯

  從Python generate_manifest.py 移植：

  | 元件             | 說明                            | 複雜度                        |
  |------------------|---------------------------------|-------------------------------|
  | PlatformDetector | 從 filename偵測 os-arch         | 中(已有類似 platform.rs)      |
  | GitHubAPI        | 取得 repo info + latest release | 低 (已有 providers/github.rs) |
  | Gist 解析        | 從 Gist URL 取得 script 資訊    | 中                            |
  | Raw script 解析  | 從 raw URL 取得 script          | 低                            |
  | Shebang 偵測     | 無副檔名時偵測 script type      | 低                            |

  3. 新格式 manifest.json (scripts with platform)

  目前 Python 版產出：
  {
    "scripts": [
      {
        "name": "rclonemm",
        "description": "...",
        "url": "...",           // 單一 URL
        "script_type": "bash",  // 單一類型
        "repo": "..."
      }
    ]
  }

  新格式（符合 Rust ScriptItem 結構）：
  {
    "scripts": [
      {
        "name": "rclonemm",
        "description": "...",
        "repo": "...",
        "platforms": {
          "bash": { "url": "https://.../rclonemm.sh" },
          "powershell": { "url": "https://.../rclonemm.ps1" }
        }
      }
    ]
  }

  4. -d 直接加入的處理邏輯

  對於 --direct 參數，需要處理：

  | 輸入類型           | 處理方式                        |
  |--------------------|---------------------------------|
  | GitHub release URL | 解析為 package，自動偵測平台    |
  | Raw script URL     | 解析為 script，偵測 script_type |
  | 本地 binary        | 需要用戶提供平台資訊或自動偵測  |
  | 本地 script        | 從副檔名/shebang 偵測 type      |

  同名不同平台合併邏輯：
  - 收集所有同名的package/script
  - 根據 platform 合併到同一個 entry的 platforms HashMap

  5. 技術複雜度評估

  | 工作項目               | 複雜度 | 預估LOC |
  |------------------------|--------|---------|
  | CLI 參數重構           | 低     | ~30     |

  | 輸出新格式             | 低     | ~30     |
  | 總計                   |        | ~520    |
