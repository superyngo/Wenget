● 現在我已經找到了問題的根本原因。讓我總結分析結果。

  問題分析

  wenget update self 破壞 bucket.json 並重設 package 資訊的根本原因是權限等級切換導致路徑變更。

  核心問題流程

  1. 權限偵測機制 (src/core/privilege.rs:17-35)
    - is_elevated() 會偵測當前進程是否以 root (Unix) 或 Administrator (Windows) 權限運行
    - 結果會被 OnceLock 快取
  2. 路徑自動切換 (src/core/paths.rs:42-55)
  pub fn new() -> Result<Self> {
      let is_system = is_elevated();// 檢查是否為 root

      let root = if is_system {
          Self::system_root_path()  // -> /opt/wenget/ (Linux)
      } else {
          Self::user_root_path()?   // -> ~/.wenget/
      };
  }
  3. 當以 sudo 執行 wenget update self 時：
    - WenPaths::new() 偵測到 root 權限
    - 路徑從 ~/.wenget/ 變為 /opt/wenget/
    - 系統讀取 /opt/wenget/buckets.json（不存在或為空）
  4. auto-repair 機制觸發 (src/bucket.rs:52-98, src/core/config.rs:52-97)
    - 當 JSON 解析失敗（包括檔案不存在時返回空配置）
    - BucketConfig::load() 發現 /opt/wenget/buckets.json 不存在 → 返回空 BucketConfig
    - Config::load_installed() 同樣返回空的 InstalledManifest
  5. 結果：
    - 所有操作都在 /opt/wenget/ 下進行
    - 使用者的 ~/.wenget/ 資料雖然還在，但 wenget 再也讀不到它們
    - 如果執行了寫入操作，可能會覆蓋 /opt/wenget/ 中的檔案

  為什麼 update self 特別容易觸發這個問題？

  因為使用者在升級 wenget 時經常需要寫入 /usr/local/bin/ 或其他系統目錄，所以會使用 sudo wenget update self。這時候：

  1. 以 root 執行 → 路徑切換到 /opt/wenget/
  2. 讀取 /opt/wenget/buckets.json（不存在）→ 返回空配置
  3. 使用者誤以為所有 bucket 都消失了

  解決方向

  update self 應該只更新執行檔本身，不應該觸發 WenPaths::new() 的權限偵測機制，或者應該使用 WenPaths::new_user() 強制使用使用者層級的路徑，因為自我更新不需要改變配置檔案的存取位置。

  問題分析

  wenget info 查詢尚未安裝的包時顯示所有平台（包含 musl 和 gnu），但安裝後變成只顯示 musl 平台的根本原因是：

  問題流程

  1. 安裝前（從 bucket cache讀取）：
    - info 命令通過 PackageResolver::resolve_from_cache() 獲取包信息
    - Cache 中的 Package.platforms 來自 bucket manifest
    - Bucket manifest 是手動維護的，包含所有平台的完整信息（musl、gnu 等）
  2. 安裝時（第775-799 行，add.rs）：
  match gh.fetch_package(repo_url) {
      Ok(latest_pkg) => {
          // 從 GitHub API 獲取最新的 package 信息
          (latest_pkg, version, false)  // using_fallback = false
      }
      // ...
  }
    - 安裝時調用 GitHubProvider::fetch_package() 從 GitHub API 獲取最新信息
    - 這會調用 BinarySelector::extract_platforms() 動態解析 release assets
  3. 安裝後更新 cache（第 844-855 行，add.rs）：
  if !using_fallback {
      packages_to_cache.push((pkg_to_install.clone(), resolved.source.clone()));
  }
  // ...
  update_cache_with_packages(config, packages_to_cache)
    - 如果成功從 GitHub API 獲取（不是 fallback），就會用 GitHub API 獲取的包信息覆蓋 cache
  4. 問題的根源（github.rs:111 和 platform.rs:745-785）：
    - GitHubProvider::fetch_package() 調用 BinarySelector::extract_platforms(&assets)
    - extract_platforms 只會提取當前 release 中存在的 assets
    - 如果某個 release 只發布了 musl 版本而沒有 gnu 版本，cache 就會被更新為只有 musl
  5. 安裝後查詢 info：
    - Cache 已被覆蓋為GitHub API 獲取的信息（只有 musl）
    - info 命令讀取 cache 時就只能看到 musl 平台

  核心問題

  update_cache_with_packages() 函數會用 GitHub API 動態解析的結果覆蓋 bucket manifest中手動維護的完整平台信息。

  這是一個設計缺陷：bucket manifest 是經過人工維護的可信數據源，不應該被動態解析的結果覆蓋。

  解決方向

  有幾個可能的修復方案：
  1. 不要更新 bucket 來源的包：如果包來自 bucket (PackageSource::Bucket)，不要用 GitHub API 的結果覆蓋 cache
  2. 合併而非覆蓋：保留 bucket manifest 中已有的平台，只添加新發現的平台
  3. 分離存儲：cache 中區分「來自 bucket」和「來自 API」的數據
